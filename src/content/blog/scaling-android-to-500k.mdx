When I joined Powerplay as the second engineering hire in April 2020, the Android app had maybe a few hundred users. By the time I left in 2022, it had crossed 500,000.

Here's what I learned.

## Architecture first, always

The decision that paid off most was betting on MVVM + Clean Architecture from day one, even when the team was just two people. It felt like over-engineering at the time. It wasn't.

When you're scaling an engineering org from 2 to 20+, the architecture is the hiring documentation. A new Android engineer who joins should be able to understand the codebase structure without asking anyone. MVVM gave us that.

```kotlin
// The pattern that scaled
class WorkOrderViewModel(
  private val repository: WorkOrderRepository
) : ViewModel() {
  val workOrders: StateFlow<List<WorkOrder>> =
    repository.observeWorkOrders()
      .stateIn(viewModelScope, SharingStarted.Lazily, emptyList())
}
```

## The 99.98% crash-free number

This didn't happen by accident. Three things made it possible:

**1. Crash reporting from day one.** We had Firebase Crashlytics wired up before the first public release. You can't fix what you can't see.

**2. Automated testing on critical paths.** Not 100% test coverage — that's a vanity metric. We tested the work order creation flow, the site visit flow, and the payment flow. Those three flows covered 80% of user sessions.

**3. CI/CD that blocked bad builds.** Every PR ran the test suite. Merging a failing build to main required a second engineer to explicitly override. That forcing function matters.

## Team scaling lessons

The engineering org growing from 2 to 20+ in 2 years is honestly harder than the technical scaling. A few things I'd do the same again:

- **Set coding standards early.** We wrote a one-page Android style guide in month 2. It prevented 90% of code review arguments before they happened.
- **Mentoring is a multiplier.** I spent probably 20% of my time on 1:1s and code reviews for the 5+ Android engineers I mentored. That time paid back 5x in velocity.
- **Hire for ownership mindset.** Skills can be taught. Whether someone actually cares if production is broken at midnight — that's a personality trait.

## What I'd do differently

One thing I'd change: we waited too long to invest in an Android component library. By the time we built it (around the 10-engineer mark), we had six different button styles across the app and it took two sprints to unify them.

Build your internal component library at 3 engineers, not 10.
